# ViMouse 修正・配布メモ（2026-06-07）

## 仕様の再確認
- ViMouse には README 上の `Input mode` と `Normal mode` がある。実装上は `Input mode` が通常入力状態、`Normal mode` が mouse mode に相当する。
- `Input mode` では通常キー入力を通す。例外は `^;` / `fn+;` の mouse mode への切替と、`^h` / `^j` / `^k` / `^l` の矢印キー化。
- mouse mode では `h` / `j` / `k` / `l` などをマウス操作として消費する。`i`、英数、かなで `Input mode` に戻る。
- 入力フックは delegate が `true` を返したキーだけを消費する。通常入力を壊さないため、この意味論を変えないことが重要。

## マルチディスプレイ境界バグ
- 旧実装は `NSEvent.mouseLocation` の AppKit 座標と `CGDisplayBounds` / `CGEvent` の CoreGraphics 座標を混在させていた。
- 特に境界外判定後の display lookup が AppKit 座標で実行され、複数ディスプレイ環境で別ディスプレイや不正な矩形へ clamp される可能性があった。
- 修正では `CGEvent(source: nil)?.location` を基準にし、移動・境界判定を CoreGraphics 座標に統一した。
- `MouseMovementBounds.nextPosition` は、移動先がいずれかの画面内ならそのまま通し、画面間の gap に入る場合だけ現在画面内に clamp する。

## テスト方針
- `MouseMovementBounds` を単体テスト可能な純粋ロジックとして分離した。
- 追加テスト:
  - 隣接ディスプレイには境界を越えて移動できる。
  - 画面間に gap がある場合は現在画面内に留まる。
  - delegate が消費しないキーは `InputHook` が透過する。
  - キーリピートは初回 key down の消費判定を維持する。
- SSH 上の `xcodebuild test` は Xcode の test manager で停止することがあるため、`test-unit.sh` は test bundle を build し、`xcrun xctest` で直接実行する。

## 配布・notarize
- 配布成果物は `dist/` に置くが Git 管理しない。
- CLI 署名時は一時 keychain を作り、Developer ID Application / Installer 証明書を import して使うと、ログイン keychain のロック状態に依存しにくい。
- app zip の配布では app を Developer ID Application で署名し、zip を notary submit して app に staple する。
- pkg インストーラーでは app を Developer ID Application で署名し、`pkgbuild` で component pkg を作り、`productbuild` で Developer ID Installer 署名を付けてから notary submit する。
- notarize 前の pkg は `spctl --type install` で `Unnotarized Developer ID` として reject される。notarize/staple 後に `source=Notarized Developer ID` になることを確認する。
